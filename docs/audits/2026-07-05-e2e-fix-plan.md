## Ground rules (do not skip)

   task, locate the code by the quoted **symbol / string anchor**, not the line number. Read the
   surrounding code and confirm the described behavior still exists before changing anything. If a
   finding no longer reproduces, note it and skip — do not force a change.
2. **Follow CLAUDE.md exactly:** Conventional Commits; all work through PRs; `strict` TS with no
   `any` (no bare `any` without an inline justification comment); validate external input with Zod;
   never `--no-verify`; never put CI jobs on macOS; preserve `package-lock.json`; don't add new
   runtime libraries — everything here is doable with what's installed.
3. **Reuse existing patterns, don't invent.** For user-facing error feedback use the existing toast:
   `import { useToast } from "@/components/kash/ui/ToastProvider"` → `const { toast } = useToast();`
   → `toast({ message: "…", variant: "error" })`. Variants are `"neutral" | "success" | "info" |

   `AbyssPromoteMenu.tsx`), match that inline style instead of a toast, for consistency within that
   component. **Caveat:** `ToastProvider` is mounted in `AppShell` only — it is NOT present on the

   judgment-heavy refactors. Rebase on `main`, branch with a `fix/…` or `feat/…` name.
5. **Verify each phase before opening the PR:** `npm run typecheck && npm run lint && npm run build`,
   then use the preview/dev server to exercise the actual flow you changed (don't just eyeball code).
   The dev server auth-bypasses in `NODE_ENV=development`, so you land straight in the app with seeded

## Phase 1 — Crash & dead-page (PR: `fix/health-500-and-error-boundaries`) — §0, §8

**1a. `/health` returns a blank 500.** `src/app/health/page.tsx` — `HealthPage` awaits
`Promise.all([caller.healthChecks.list(...), caller.healthChecks.getLatest(...)])` with no try/catch,
and there is no route error boundary, so any DB error white-screens the page.

- Add `src/app/health/error.tsx` (a `"use client"` error boundary with a reset button and a link
  back to `/today`) so a thrown query degrades gracefully instead of blank-screening.
- Also wrap the two caller queries so a DB failure renders the page's existing "No health checks…"
  / error state rather than throwing. Keep the existing empty-states.

**1b. No app-wide error/not-found boundaries.** Only `src/app/global-error.tsx` exists.

- Add a top-level `src/app/error.tsx` and `src/app/not-found.tsx` styled with the app shell/tokens,
  so server-component throws and `notFound()` (e.g. a well-formed but missing `/projects/{uuid}`,
  §6) render inside app chrome with a way back — not the bare Next.js default pages.

**1c. (DECISION — ask the user first) `/health` is public + unlinked.** It is not in
`PROTECTED_PREFIXES` (`src/lib/auth/paths.ts`) and its tRPC procedures use `baseProcedure` (no auth).
Ask: _is `/health` an internal ops page (then gate it behind auth / a flag and leave it out of nav)
or user-facing (then add a nav/command-palette entry)?_ Implement per their answer. Do **not** change
auth config or `supabase/config.toml` (see CLAUDE.md — sign-up must stay off; don't touch auth sync).

> Note: the audit observed the DB connection pool exhausting (`PostgresError 53300`). If that recurs
> under normal use it's a connection-lifecycle issue beyond this page — flag it to the user; don't
> chase it as part of this PR unless they ask.

## Phase 3 — Dead controls (PR: `fix/dead-controls`) — §2, §3, §4

**3a. This Week lens Group/Filter/Tag chips do nothing.** `LensControlBar` (rendered from
`src/components/kash/week/ThisWeekSurface.tsx`, `scope="this-week"`) toggles `state.group` /
`state.filters` / `tagFilter`, but `src/components/kash/plan/week/WeekCanvas.tsx` maps `partitioned`
tasks to columns without ever consulting those. Either (a) apply the group/filter/tag state to the
week column mapping in `WeekCanvas`, or (b) if week-scoped lensing isn't intended, hide the chips for
`scope="this-week"`. **Ask the user which** — (a) is more work but is the implied intent. Confirm how
the Today surface consumes `lens-apply` (`BucketSection`/`LensGroupSection`) and mirror that.

**3b. Plan month-header "zoom to month" button is dead for non-Quarter callers.**
`src/components/kash/plan/quarter/MonthColumn.tsx` — the header button calls `onZoomMonth?.()` which
is `undefined` for common callers, so it renders as an interactive affordance that no-ops. Either
wire `onZoomMonth` through the callers that render `MonthColumn`, or render the header as static
(no button, no `hover:underline`, no "zoom to month" aria-label) when no handler is passed.

**3c. Onboarding "Pull from a project" is dead.** `src/components/kash/.../OnboardingModal.tsx`
mounts `MorningHandoffModal` with `projects={[]}` and `onPullProjectTask={() => undefined}`. Either
pass the real projects list + a working pull handler, or hide the "Pull from a project" section
during onboarding so first-run users don't see a non-functional control.

**3d. Bingo "Link existing task ID…" requires a hand-typed UUID.**
`src/components/kash/plan/bingo/BingoGoalPanel.tsx` (anchor `Link existing task ID…`, state
`linkTaskId`). Replace the raw text field with a task picker/search (reuse the project/task list the
panel already has access to), and add `onError` feedback to `linkTaskMutation` so a bad link is
visible instead of clearing optimistically and no-oping.

## Phase 4 — Destructive-action & data-staleness safety (PR: `fix/destructive-and-stale-data`) — §3, §6

**4a. Slip/replan leaves task data stale.** `src/components/kash/projects/ProjectSlipReplanCard.tsx`
— on apply it invalidates `phases.listByProject` but not `tasks.listByProject`. Add the
`tasks.listByProject` invalidation (scoped to the project). Add `onError` feedback. Reconsider the
auto-dismiss so the card can reappear if a refetch shows the slip persists (don't dismiss on error).

**4b. `ConfirmDialog` fires destructive action on Enter.**
`src/components/kash/projects/ConfirmDialog.tsx` — Enter triggers `onConfirm()` while the confirm
button is auto-focused, so pressing Enter right after opening a delete dialog deletes immediately. For
destructive variants, either don't auto-confirm on Enter, or require the confirm button to be
explicitly focused/clicked (or a typed confirmation). Keep non-destructive dialogs as-is.

**4c. Projects have no delete/archive UI.** A `projects.delete` mutation exists
(`src/trpc/routers/projects.ts`, anchor `delete`) but `src/components/kash/projects/ProjectMenu.tsx`
exposes only Import history / Save as template. Add a Delete (or Archive) item to `ProjectMenu`
behind the existing `ConfirmDialog` (destructive copy), wired to `projects.delete`, with
`onError` feedback and navigation back to `/projects` on success. **Confirm with the user** whether
they want hard delete or archive.

**4d. New project doesn't navigate into itself.**
`src/components/kash/projects/NewProjectForm.tsx` calls `onCreated(project.id)`, but
`ProjectsIndex.tsx` passes `onCreated={() => setFormOpen(false)}` and drops the id. Change the index
to `router.push(\`/projects/${id}\`)` on create so the user lands in the new workspace.

## Phase 5 — Silent-failure sweep (PR: `fix/mutation-error-feedback`) — §0 (biggest leverage)

Add `onError` user feedback (toast `variant:"error"`, or inline `isError` where the component already
uses that pattern) to these mutations. Where the audit also flags **no optimistic UI on a slow
network**, at minimum ensure the acted-on control shows a pending/disabled state; full optimistic
updates are optional and can be a follow-up. Do NOT swallow errors; remove any empty `catch {}`.

  `src/hooks/useEssentialNudges.ts` — `goal_step_add` / `balance_add` run inside `void(async…)()`
  with `catch {}` then always `dismiss()`. Only dismiss on success; on failure surface an error and
  keep the chip. Also make dismissals persist (they're a session-only `useState<Set>`; mirror the
  EoW pattern in `useEowReviewTrigger.ts` which persists via `eow-storage`).
- **5b. Chat:** `src/hooks/useChatPanel.ts` — `applyProposal`/`dismissProposal` have no try/catch

  is set during `appendUserMutation.mutateAsync` — set an abort/cancel flag that the send path checks.
  `src/hooks/useChatSuggestions.ts` — the `work_on` branch has no `catch`; add one that sets
  `streamError`. Add a streaming indicator before the first token (`streamingText === ""`) so the
  panel isn't frozen-empty, and a Retry affordance on `streamError` in `ChatRail.tsx`.
- **5c. Plan:** add `onError` to the TaskRow delete/skip (`TaskRow.tsx`), and the WeekCanvas /

  `BalancePassGhosts`, `WeekDraftGhosts`) don't let one rejection abort the rest silently — catch

- **5d. Abyss/Backlog:** `AbyssList.tsx` Row `deleteMutation` + `promoteToday`, `abyss.create` in

## Phase 6 — Autocomplete parity (DECISION REQUIRED — ask before coding) — §0, §11

The audit's headline consistency issue. Two facts to confirm with the user first:

- There are **two** assist engines: `src/lib/parser/composer-assist.ts` (Plan/Today/Week/Onboarding
  via `QuickInput`) and `src/lib/parser/project-composer-assist.ts` (Projects via `NewItemRow`).
  The recent "project slug in any semicolon segment" fix landed only in the former, so they drift.
- Many task-property inputs are plain text with no completion: inline title edit (`TaskRow.tsx`),
  occurrence edit (`OccurrenceMenu.tsx`), backlog capture (`AbyssComposer.tsx`), Bingo add
  (`BingoQuickAdd.tsx`), month intentions (`MonthIntentionsEditor.tsx`), quarter theme
  (`QuarterThemePicker.tsx`), task-detail category (`TaskDetail.tsx`) & tags (`TaskTagsEditor.tsx`),
  morning "Pull from a project" search (`MorningHandoffModal.tsx`).

**Ask the user to choose scope** (don't wire all of them blindly — it's a large surface and some may
be intentionally plain):

- (A) Unify the two engines so they can't drift (highest value, contained), and/or
- (B) Extend the composer assist to a chosen subset of the plain inputs (e.g. inline title edit +
  backlog capture, which are the most jarring), and/or
- (C) Deepen the assist model itself: it's currently ghost-text + Tab-accept, top-1 match only — no
  arrow-nav, no clickable list, no Escape-to-dismiss, no "no match" hint, and an empty segment ghosts
  `projects[0]` you never typed. Decide whether to add a small suggestion dropdown.

Implement only the option(s) they pick, one PR per coherent chunk.
