Where does the suggested catalog physically live?

Your own practices always live in a table (care_activities). The only question is the ~23 suggested seeds: a code list, or their own DB table.

OPTION A

Static seed in code

// src/lib/care/seed-catalog.ts
export const CATALOG = [
  { title: "10-minute walk", theme: "move" },
  { title: "Drink a glass of water", theme: "nourish" },
  // …~23 entries
];
Adopt → copies the entry into your care_activities row (source:"suggested"). The catalog itself never hits the DB.
+ One table total; no catalog migration/RLS
+ Edit seeds in code, ships with the app
+ Perfect for a curated, single-user list
– Catalog changes need an app deploy
– Reworded seeds don't update already-adopted copies
OPTION B

DB catalog table

// a global table, no user_id
care_catalog (
  id, title, theme, kind
)
care_activities (
  …, catalog_id → ref
)
Adopt → inserts a care_activities row referencing the catalog id. Catalog is real data you can edit without shipping code.
+ Edit catalog without a deploy (admin/SQL)
+ Room to grow: localize, A/B, share sets
– Extra table + read-RLS for a global table
– Seeded via migration; more moving parts
– Overkill for ~23 static, personal entries

Reversible either way: you can promote a code list into a table later without changing how care_activities works. So this is a "right-sized for now" call, not a one-way door.
