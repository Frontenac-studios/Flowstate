Projects §9 · Thread — How is % progress calculated?

One project, the same state, four metrics. Watch the headline number swing.

“Book launch” — 2 big tasks (★, one done), 8 small tasks (6 done). Big tasks ≈ most of the real work. Est. hours: bigs 20h each, smalls 1h each (40h done of 48h… or is it?).
★ Write manuscript ★ Edit & proof cover briefblurbISBN author biopick printerset price launch emailstore listing
1 · Simple task count

completed ÷ total tasks

70%
7 of 10 tasks done
Treats every task as equal.
  • Dead simple; no extra data
  • Overstates — the big “Edit” task is most of the work
2 · Weighted task count · matches §2

completed weight ÷ total weight

52%
big=3, small=1 → (3+6) ÷ (6+8) = 9/14
Uses the same Top-3 / large-small weighting we locked for balance. Big unfinished work pulls the number down honestly.
  • Honest without needing time data
  • One metric across the app
  • Weights are coarse (just large/small)
3 · Time-based

hours done ÷ estimated total

46%
(20h + 6×1h) ÷ 48h est.
Truest effort picture — but leans on duration estimates, which are silent until n≥3 (Q2), so it’s blank early.
  • Most accurate when data exists
  • Unavailable early; depends on estimates
4 · Phase-based

completed phases ÷ total phases

50%
1 of 2 phases (Write ✓ · Launch ◻)
Coarse milestone view.
  • Maps to the phase structure; intuitive
  • Jumpy (0→50→100); ignores within-phase progress

Note: whichever is chosen can roll up the same way at phase level (for the Miller/timeline views) and project level (cards + §7.6 weekly review).
